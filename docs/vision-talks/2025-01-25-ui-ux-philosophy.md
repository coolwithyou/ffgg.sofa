## 2. 감정 설계

### 사용자가 오기 전 상태

```
"같은 질문에 백번째 답변하느라 지쳤다"
"챗봇? 어려워 보여서 엄두가 안 났다"
"시간도 없고 돈도 없는데 뭔가 해결책이 필요하다"
```

### 첫 3초에 느꼈으면 하는 감정

```
"오, 이건 나도 할 수 있겠다"
"깔끔하네, 복잡하지 않네"
"뭔가 친근하고 부담 없다"
```

### 사용 후 느꼈으면 하는 감정

```
"진짜 5분 만에 됐네"
"이게 끝이야? 쉽다"
"이거 다른 사람한테도 알려줘야겠다"
```

## 3. UI 원칙

### 원칙 1: 선택지는 최대 3개

```
❌ 버튼 5개가 나열된 화면
⭕ 버튼 2개 + "더 보기"
```

사용자에게 결정 피로를 주지 않는다.
한 화면에서 할 수 있는 일은 명확하게 제한한다.

### 원칙 2: 다음 할 일이 항상 보인다

```
❌ "대시보드에 오신 것을 환영합니다"
⭕ "가이드를 만들어 시작하세요 [시작하기]"
```

RPG 게임처럼 퀘스트가 명확하다.
"뭘 해야 하지?"라는 순간이 없다.

### 원칙 3: 깊이는 1단계까지

```
❌ 메뉴 → 서브메뉴 → 서브서브메뉴 → 설정
⭕ 메뉴 → 바로 기능
```

모든 기능은 2클릭 이내에 도달한다.
복잡한 구조는 사용자가 아닌 시스템이 감당한다.

### 원칙 4: 전문 용어 없음

```
❌ "데이터셋을 생성하고 청크를 임베딩합니다"
⭕ "파일 올리면 끝. AI가 알아서 정리해요"
```

개발자만 아는 단어는 절대 노출하지 않는다.
초등학생도 이해할 수 있는 말로 쓴다.

### 원칙 5: 여백은 자신감

```
❌ 빽빽하게 채운 화면
⭕ 숨 쉴 공간이 있는 화면
```

여백은 비어있는 게 아니라 의도된 것이다.
"우리는 이것만 잘해"라는 메시지다.

## 4. 비주얼 원칙

### 컬러 철학

**컬러는 기능을 말한다**

| 역할 | 컬러 | 용도 |
|------|------|------|
| Primary | 따뜻한 포인트 (오렌지/코랄 계열) | CTA, 주요 액션 |
| Secondary | 차분한 블루/그레이 | 정보, 보조 요소 |
| Success | 그린 | 완료, 성공 |
| Warning | 옐로우 | 주의, 안내 |
| Background | 화이트/오프화이트 | 기본 배경 |
| Surface | 라이트 그레이 | 카드, 섹션 구분 |

**원칙**:
- 포인트 컬러는 1개만 강하게
- 나머지는 뉴트럴로 받침
- 컬러가 의미 없이 쓰이지 않음

### 타이포그래피

**Pretendard Variable 단일 사용**

| 용도 | 스타일 | 크기 |
|------|--------|------|
| 대제목 | Bold | 32-48px |
| 중제목 | SemiBold | 24px |
| 본문 | Regular | 16px |
| 보조 | Regular | 14px |
| 캡션 | Medium | 12px |

**원칙**:
- 폰트는 하나로 통일 (피로감 방지)
- 문서를 다루는 서비스이므로 장시간 가독성 중요
- 웨이트와 크기로 위계 표현

### 형태 언어

**둥글고 부드러운 형태**

| 요소 | 스타일 |
|------|--------|
| 버튼 | border-radius: 12px (또는 full) |
| 카드 | border-radius: 16px |
| 입력창 | border-radius: 8px |
| 아이콘 | 라운드 스타일, 2px 스트로크 |

**원칙**:
- 날카로운 모서리 지양
- 부드러운 인상 = 접근하기 쉬운 느낌
- 그림자는 최소화, 쓸 때는 부드럽게

### 간격 시스템

**8px 그리드 기반**

| 토큰 | 값 | 용도 |
|------|-----|------|
| space-xs | 4px | 아이콘과 텍스트 사이 |
| space-sm | 8px | 관련 요소 간격 |
| space-md | 16px | 그룹 내 간격 |
| space-lg | 24px | 섹션 내 간격 |
| space-xl | 32px | 섹션 간 간격 |
| space-2xl | 48px | 대영역 간격 |

## 5. 인터랙션 원칙

### 피드백은 즉각적으로

```
버튼 클릭 → 즉시 상태 변화 (로딩, 비활성화)
성공 → 명확한 완료 표시
실패 → 친절한 에러 메시지 + 해결 방법
```

### 애니메이션은 의미 있게

```
❌ 화려한 효과를 위한 애니메이션
⭕ 상태 변화를 이해시키는 애니메이션
```

| 상황 | 애니메이션 |
|------|-----------|
| 화면 전환 | 부드러운 페이드 (150-200ms) |
| 요소 등장 | 살짝 위로 올라오며 페이드 인 |
| 호버 | 미세한 확대 또는 색상 변화 |
| 로딩 | 스켈레톤 또는 진행 표시 |

### 에러는 친절하게

```
❌ "Error: Invalid input"
⭕ "이메일 형식이 올바르지 않아요. @를 포함해주세요"
```

에러 메시지도 브랜드 톤을 유지한다.
기술적 용어 대신 일상 언어로 안내한다.

## 6. 상태 기반 UI

### 핵심 개념

**화면은 "현재 상태"에 따라 달라진다**

사용자가 메뉴를 탐색하는 게 아니라,
시스템이 현재 상황에 맞는 화면을 보여준다.

### 상태 정의

| 상태 | 조건 | 보여줄 화면 |
|------|------|------------|
| Empty | 가이드 0개 | 시작 유도 화면 |
| Processing | 가이드 생성 중 | 진행 상태 화면 |
| Ready | 가이드 있음, 미공개 | 공개 유도 화면 |
| Live | 공개됨 | 대시보드 화면 |

## 8. 컴포넌트 원칙

### 버튼

| 타입 | 용도 | 스타일 |
|------|------|--------|
| Primary | 주요 액션 (1개만) | 포인트 컬러, 채움 |
| Secondary | 보조 액션 | 테두리만 또는 연한 배경 |
| Ghost | 덜 중요한 액션 | 텍스트만 |

```
✅ 한 화면에 Primary 버튼은 1개
✅ 버튼 텍스트는 동사로 ("만들기", "공개하기")
❌ "확인", "제출" 같은 모호한 텍스트
```

### 카드

```
┌────────────────────────────────┐
│  ┌────┐                        │
│  │ 📄 │  제목                   │
│  └────┘  설명 텍스트            │
│                                │
│                    [액션 버튼]  │
└────────────────────────────────┘
```

- 충분한 패딩 (24px 이상)
- 명확한 위계 (아이콘 → 제목 → 설명 → 액션)
- 호버 시 미세한 상승 효과

### 입력 필드

```
라벨
┌─────────────────────────────────┐
│ placeholder...                  │
└─────────────────────────────────┘
도움말 텍스트
```

- 라벨은 항상 위에
- placeholder는 예시 (형식 안내 아님)
- 도움말은 필요할 때만
- 에러 시 빨간 테두리 + 메시지

## 9. 반응형 원칙

### 브레이크포인트

| 이름 | 범위 | 레이아웃 |
|------|------|----------|
| Mobile | ~640px | 1열, 풀스크린 |
| Tablet | 641~1024px | 적응형 |
| Desktop | 1025px~ | 사이드바 + 콘텐츠 |

### 모바일 우선 고려사항

- 터치 타겟 최소 44x44px
- 중요한 액션은 엄지 닿는 위치에
- 스크롤은 세로만 (가로 스크롤 지양)
- 2열 UI는 모바일에서 탭 전환으로

## 10. 접근성 원칙

### 기본 준수 사항

- 명암비 4.5:1 이상 (WCAG AA)
- 키보드로 모든 기능 접근 가능
- 스크린 리더 호환 (적절한 aria 레이블)
- 포커스 상태 명확히 표시

### 폰트 크기

- 최소 본문 크기: 16px
- 모바일에서도 확대 가능하게

## 11. "조잡하지 않은 심플"을 위한 체크리스트

### 디자인 리뷰 시 확인

- [ ] 요소 간 간격이 8px 그리드를 따르는가?
- [ ] 컬러가 의미 없이 사용되진 않았는가?
- [ ] 한 화면에 Primary 버튼이 2개 이상 있진 않은가?
- [ ] 전문 용어가 노출되진 않았는가?
- [ ] 다음 할 일이 명확한가?
- [ ] 여백이 충분한가?
- [ ] 모바일에서도 터치하기 편한가?

### 개발 리뷰 시 확인

- [ ] 애니메이션 duration이 150-300ms 사이인가?
- [ ] 로딩 상태가 있는가?
- [ ] 에러 메시지가 친절한가?
- [ ] 키보드 네비게이션이 되는가?
