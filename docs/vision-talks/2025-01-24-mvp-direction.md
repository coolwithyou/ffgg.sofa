## 1. 서비스 비전 정리

### 핵심 포지셔닝

**"복잡한 제품/서비스를 파는 소규모 기업을 위한, 설정이 극도로 쉬운 FAQ 가이드"**

### 브랜딩 결정 ✅

- **서비스명**: faq.guide (도메인이 곧 브랜드)
- **도메인**: faq.guide (Cloudflare에서 확보 완료)
- **고객 페이지 URL**: faq.guide/[회사명] (예: faq.guide/ottchil)
- **관리 화면**: Studio
- **상세 브랜딩 전략**: [2025-01-25-branding-strategy.md](./2025-01-25-branding-strategy.md)

### 기존 CS 챗봇(채널톡 등)과의 차별점

| 구분 | 기존 CS 챗봇 | 우리 서비스 |
|------|-------------|-------------|
| 목적 | 컴플레인 처리/상담 | 제품 학습/가이드 |
| 역할 | 응대자 | 퍼스널 쇼퍼/도슨트 |
| UI | 우하단 작은 창 | 전면 2열 레이아웃 (문서 + 대화) |
| 타겟 | 대기업 CS팀 | 자원 없는 소규모 기업 |

### 우리 챗봇의 핵심 역할

- 사용자가 "궁금한 걸 알려주는" 것이 아니라 **"뭘 궁금해해야 하는지 찾게 해주는"** 역할
- 복잡한 제품은 고객이 "뭘 모르는지 모르는" 상태가 많음
- 응대가 아닌 **가이드/도슨트/퍼스널 쇼퍼** 역할

### 비주얼 컨셉

- 2열 다이어로그: 좌측에 참조 문서, 우측에 챗봇 대화
- 대화하면서 관련 문서가 실시간으로 표시
- 고객이 매뉴얼을 더 친숙하게 접할 수 있는 경험

## 2. 논의 과정에서 검증한 가정들

### 검증이 필요했던 질문들

1. **"복잡한 제품을 파는 소규모 기업"이 충분히 존재하는가?**
   - 확인된 케이스: 옻칠 기업
   - 잠재 케이스: 아트페어 운영사 (수백 개 갤러리, 도록, 행사 CS)

2. **"매뉴얼이 이미 존재한다"는 전제가 맞는가?**
   - 체계적 매뉴얼은 없더라도 제품 상세페이지, 카탈로그, 커뮤니티 QnA, CS 히스토리 등은 존재
   - 이것들을 합쳐서 AI 학습용 QA 문서로 만드는 것이 우리의 강점이 될 수 있음

3. **2열 UI가 모바일에서도 작동하는가?**
   - 추가 고민 필요 (미해결)

## 3. 전략적 선택지 분석

### 두 가지 방향

| | 화이트 글러브 | 캐주얼 셀프서비스 |
|---|---|---|
| 차별점 | 명확함 (고품질 문서화) | 불명확함 (가격? UI?) |
| 스케일 | 안 됨 | 됨 |
| 운영자 시간 | 많이 듬 | 거의 안 듬 |
| 경쟁 | 적음 | 많음 (Chatbase 등) |
| 수익성 | 건당 높음 | 건당 낮음 (볼륨 필요) |

### 실제 작업 경험 (옻칠 케이스)

- 입력: PDF, HWP 문서
- 과정: NotebookLM + Claude Code로 여러 번에 걸쳐 마크다운 편집
- 소요 시간: 약 2시간
- 결론: "쉬운 셀프서비스"와 "고품질 마크다운 변환"은 현재 기술로 양립 어려움

## 4. 발견한 핵심 인사이트

### 진짜 문제 정의

> "배울 건 많은데, 누가 그걸 다 읽겠어. 1:1 코칭이 있으면 좋은데 비싸고 없고."

이 패턴이 반복되는 영역:
- 취미/공예: 가죽공예, 도자기, 뜨개질, 레진아트
- 전문 장비: 커피 머신, 오디오, 3D 프린터
- 건강/웰니스: 필라테스 기구, 보충제
- 반려동물: 특수 사료, 질병 관리

**공통점**: "관심은 있는데 진입장벽이 높은" 영역

### 한국 시장 특수성

- 대부분의 한국 사람들은 챗봇에 대한 신뢰가 낮음
- 카카오톡 메신저로 "사람"에게 CS 요청하는 건 익숙
- 챗봇은 답답해함 → **이게 곧 풀어야 할 문제이자 시장**

## 5. 결론: MVP 방향

### 핵심 전략

**"옻칠로 먼저 증명하고, 검증되면 유사 영역으로 확장"**

### 첫 번째 타겟

- 옻칠 전문 기업 (본인 비즈니스)

### MVP 플로우

1. 문서를 하나씩 넣으면 마크다운으로 변환
2. 마크다운들을 모아 대주제/분량 단위로 "위키" 페이지 구성
3. 위키 페이지를 근거로 챗봇이 사용자 질문에 응답

### 검증 항목

1. 고객이 실제로 챗봇을 쓰는가?
2. 뭘 물어보는가?
3. "이거 좋네요" vs "그냥 카톡 주세요" 반응은?
4. CS 문의가 줄었는가?
5. 구매 전환에 영향이 있는가?

### 향후 확장 계획

검증 성공 시:
- 유사 영역 (취미/공예, 전문 장비 등)으로 확장
- 캐주얼 티어 추가 (URL 입력 → 자동 챗봇 생성)
- 무료 티어로 리드 생성 → 프리미엄(화이트 글러브)으로 전환

## 9. FAQ vs 가이드북 논의

### 핵심 질문

"입력 형식을 FAQ(Q&A 쌍)로 하면 AI 답변이 제한되지 않을까?"

### 결론: 입력은 FAQ, 내부는 연결된 지식

**사용자가 보는 화면 (입력)**
```
Q: 옻칠 붓은 어떻게 관리하나요?
A: 사용 후 즉시 기름으로 닦아주세요...

Q: 초보자용 옻은 뭐가 좋아요?
A: 정제 옻을 추천합니다...
```

**AI가 학습하는 것 (내부)**
```
"옻칠 도구 관리에 대한 지식: 붓 관리는...
초보자 추천 재료는... 이 둘은 연관되어 있으며..."
```

### 왜 이게 가능한가?

- FAQ 입력 → 청크로 저장 → 임베딩
- 질문이 들어오면 **의미 기반 검색**으로 관련 청크 조합
- Q1에 대해 A1만 답하는 게 아니라, 연관된 A2, A3도 참조 가능

### 두 가지 진입점 유지

| 경로 | 사용자 상황 | 액션 |
|------|-------------|------|
| **Q&A 직접 작성** | "자주 받는 질문들이 있어" | 직접 입력 |
| **파일 업로드** | "PDF 매뉴얼이 있어" | 업로드 → AI가 Q&A 추출 |

## 10. 브랜딩: faq.guide

### 왜 faq.guide인가?

1. **즉시 이해됨**: "FAQ 가이드" - 설명 필요 없음
2. **도메인이 곧 브랜드**: faq.guide/ottchil
3. **용어 체계와 일치**: 가이드북, 가이드 만들기 등
4. **친숙한 URL 패턴**: instagram.com/username과 같은 구조

### URL 구조

```
faq.guide/[회사명]          → 공개 페이지
faq.guide/studio            → 관리 화면 (로그인 필요)
faq.guide/studio/[회사명]   → 특정 가이드 편집
```

### .guide 도메인에 대해

- 인지도는 .com보다 낮지만
- "faq.guide/ottchil" 전체 맥락에서 자연스럽게 이해됨
- 오히려 기억에 남을 수 있음
